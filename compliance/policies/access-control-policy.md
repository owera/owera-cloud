# Access control policy

**Scope:** every authentication boundary in the Owera stack — cloud plane (api, web, status), operator plane (gateway, workers), supporting SaaS (Stripe, Cloudflare, Fly, Vercel, GitHub, 1Password), and customer-facing tenants.

## 1. Principle

Principle of least privilege, default-deny. Every role gets the minimum access required for its function; access expires on role change; access is auditable.

## 2. Identity sources

| Boundary | Identity provider | MFA required |
|----------|-------------------|--------------|
| Owera personnel (all SaaS) | Google Workspace SSO + WebAuthn | Yes — hardware key preferred, TOTP acceptable, SMS forbidden |
| Customer-facing (app.owera.ai) | Clerk *or* WorkOS (TBD; see [vendor-management-policy.md](./vendor-management-policy.md)) | Optional but offered; enforced for enterprise tier |
| Operator plane (gateway SSH) | ed25519 keypairs, passphrase-protected | Yes — passphrase + key (something-you-know + something-you-have) |
| Cloud-plane services (api → operator plane) | minisign-signed JWS over Cloudflare Named Tunnel | Cryptographic — keypair lives only in Fly secrets / macOS Keychain |

## 3. Roles (cloud plane)

| Role | Permitted | Forbidden |
|------|-----------|-----------|
| **Customer** | Their own tenant data, their own billing, their own jobs | Other tenants, system metadata, audit logs |
| **Support** | Read-only view of any tenant, audit-logged | Mutating actions, secret values, financial data beyond invoice line items |
| **Engineer** | Read/write on staging, read on prod, audit-logged | Direct prod mutations (must go via deploy pipeline) |
| **SRE** | Production deploy via pipeline; emergency break-glass into prod with logging | Read of customer payloads except during sanctioned incident response |
| **CISO** | Audit-log read; emergency revoke of any access; cannot read customer payloads without dual-approval | Routine customer-payload access |
| **CFO** | Billing data, Stripe console, financial reports | Customer payloads, source code (read-only access to docs is fine) |

## 4. Privilege escalation (break-glass)

Production access for engineers and SRE is gated by a **break-glass** procedure:

1. Engineer files a justification in `#incidents` Slack with the customer / system affected.
2. Second SRE or CISO approves in-thread (two-person rule).
3. Access is granted for 4 hours via a Fly + Cloudflare time-limited token.
4. Every action during the window is recorded to the audit-log WORM store.
5. On window expiry, access auto-revokes; an after-action note is added to the incident.

Direct production console access without break-glass is a Sev2 security incident.

## 5. Access reviews

| Cadence | Reviewer | Scope |
|---------|----------|-------|
| Quarterly | CISO | All non-customer SSO grants across SaaS (Google Workspace, 1Password, Fly, Vercel, Cloudflare, GitHub) |
| Quarterly | SRE Lead | SSH keys authorized on gateway and workers |
| Annually | CISO + CFO | Stripe console grants |
| On offboarding (≤24h) | SRE + CISO | All access revoked across all systems |

Each review produces an artifact in `audit-controls/` referencing the review date and the list of changes made.

## 6. Authentication mechanics

- Passwords: where unavoidable (legacy SaaS), 16+ chars, generated by 1Password, unique per service.
- MFA: WebAuthn / hardware key preferred. TOTP via 1Password acceptable. SMS forbidden as second factor.
- SSH keys: ed25519 only; RSA forbidden for new keys. Private keys passphrase-protected and stored in macOS Keychain.
- API tokens: scoped tokens preferred. Where a service offers only account-wide tokens (e.g. Cloudflare's legacy global API key), the token is treated as Restricted and rotated quarterly.
- Service-to-service: cryptographic identity (minisign / ed25519) over Cloudflare Named Tunnel.

## 7. Session management

- Customer sessions: 24h max, sliding refresh; logout invalidates server-side.
- Engineer SSO sessions: 8h max, then re-authenticate with MFA.
- Break-glass sessions: 4h max, non-extensible without a fresh approval.

## 8. Audit logging

Every authentication event, every privilege escalation, every customer-payload access is logged to the WORM audit store (`api/internal/audit/`). Retention: 7 years (see [`data-retention-policy.md`](./data-retention-policy.md) §3). The log schema is documented in `audit-controls/soc2-cc.yaml` under CC6.1.

## 9. On-call rotation

The on-call rotation list is the operational realization of "who can be paged into a break-glass". Maintained in PagerDuty; mirrored as a runbook in [`../runbooks/on-call-runbook.md`](../runbooks/on-call-runbook.md).
